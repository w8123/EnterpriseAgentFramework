# 后端逻辑边界与命名重塑

## 定位

本轮重塑采用一句话原则：

**不先拆服务，先拆职责；不先重命名表，先重命名边界。**

ReachAI 当前已经不是单纯的 Agent 服务、Skill 服务和 Model 服务三段式系统，而是面向 Java 企业系统的 AI 能力中台。后端架构需要先在文档、代码归属、前端说明和后续新增代码规则上收敛到稳定逻辑域，再决定是否做真实微服务拆分。

本轮目标是解决两个问题：

- 代码维护视角：避免继续把注册中心、能力资产、扫描、语义、Workflow Runtime、开放协议和知识检索都塞进模糊的 Agent / Skill 概念里。
- 产品表达视角：默认使用 `Capability / 能力`，把 `Skill` 限定为历史 SQL、API、运行时契约或兼容命名。

本轮不追求立刻物理拆服务，不改变现有端口、路由、页面布局、SQL 表名或前端代理。

## 当前部署单元

当前运行时部署仍保持三个 Spring Boot 应用：

| 部署单元 | 默认端口 | 当前路由前缀 | 当前职责 |
| --- | --- | --- | --- |
| `ai-model-service` | `18601` | `/model` | 模型实例、Chat、Embedding、Rerank、OpenAI 兼容代理 |
| `ai-skills-service` | `18602` | `/ai` | 知识库、文档入库、RAG、业务索引、向量检索，以及历史扫描器实现 |
| `ai-agent-service` | `18603` | `/api`、`/embed` 等 | Agent 入口、Workflow、Runtime、注册中心、能力资产、身份、开放协议、治理和运行观测 |

管理端开发代理继续保持：

- `/ai` -> `http://localhost:18602`
- `/api` -> `http://localhost:18603`
- `/model` -> `http://localhost:18601`

## 五个长期逻辑域

长期架构按五个逻辑域表达，而不是按当前三服务名称表达。

| 逻辑域 | 当前主要承载 | 长期名称 | 主要职责 |
| --- | --- | --- | --- |
| Model Gateway | `ai-model-service` | `reachai-model-service` | 模型实例、供应商适配、Chat、Embedding、Rerank、OpenAI 兼容代理、模型连通性测试 |
| Knowledge / Retrieval | `ai-skills-service` | `reachai-knowledge-service` 或 `reachai-retrieval-service` | 知识库、文件、chunk、文档 pipeline、RAG、向量检索、业务语义索引 |
| Capability Catalog | `ai-agent-service` 内部边界 | `reachai-capability-service` | 项目注册、实例心跳、能力快照、字段级 diff、评审 apply/ignore、扫描目录、语义文档、Tool/Capability 资产 |
| Runtime Host | `ai-agent-service` 内部边界 | `reachai-runtime-service` | Agent 入口、Workflow、GraphSpec 执行、调试、人工交互、Runtime Adapter、嵌入式执行 |
| Platform Control | `ai-agent-service` 内部边界 | `reachai-control-service` | 平台身份、RBAC、租户/项目治理、ACL、Guard、Gateway、MCP、A2A、市场、RunOps/Trace 管理面 |

这五个域是代码归属和新增功能决策的主边界。当前部署名只是运行形态，不再作为产品架构主语。

## ai-agent-service 内部边界

`ai-agent-service` 仍是当前最大部署单元，但新增代码和后续迁移应按下表归属：

| 内部边界 | 典型包/文件 | 归属规则 |
| --- | --- | --- |
| Capability Catalog | `registry/`、`scan/`、`semantic/`、`tools/definition/`、`capability/`、`graph/ApiGraph*` | 任何“发现、描述、评审、沉淀、检索企业能力资产”的代码归到这里 |
| Runtime Host | `workflow/`、`runtime/`、`agentscope/`、`skill/interactive/`、`debug/` | 任何“执行 GraphSpec、调度 Runtime、挂起恢复、调试运行”的代码归到这里 |
| Platform Control | `identity/`、`platform/`、`acl/`、`governance/`、`mcp/`、`a2a/`、`market/` | 任何“身份、权限、开放协议、治理策略、资产市场”的代码归到这里 |
| Observability | `trace/`、`runops/`、`tool_call_log` 相关服务 | 运行观测先作为 Runtime 与 Platform 的共享边界，后续再评估独立服务 |
| Context Governance | `context/` | 保持独立边界；不并入 Knowledge/Retrieval，也不并入短期 ConversationMemory |

新增 Controller 和 Service 时，应先判断业务语义归属，再决定包名和文档入口。不要因为部署在 `ai-agent-service` 就继续扩大“Agent”概念。

## ai-skills-service 命名收敛

`ai-skills-service` 的 artifactId 和目录名暂时保留，避免影响构建脚本、代理、部署和文档链接。但产品与文档口径改为：

- 当前部署单元：`ai-skills-service`
- 当前职责名：Knowledge / Retrieval deployment unit
- 长期目标名：`reachai-knowledge-service` 或 `reachai-retrieval-service`

该服务不再被描述为“技能服务”。它承载的是知识、检索、文档入库、RAG、业务索引和历史扫描器实现。

## Capability 与 Skill 命名规则

默认规则：

- 产品、文档、UI 和新功能说明使用 `Capability / 能力`。
- `Skill` 只在解释历史代码、旧表名、旧枚举、运行时契约或兼容 API 时出现。
- 新增能力资产入口优先使用 `/api/capabilities`、Capability Catalog、组合能力、交互能力等表达。

本轮明确不改：

| 保留项 | 原因 |
| --- | --- |
| `skill_draft`、`skill_eval_snapshot`、`skill_interaction` 等表名 | 活跃 SQL/Mapper/页面仍使用，贸然改名会扩大风险 |
| `skill_name`、`skill_kind` 等字段 | 属于兼容敏感存储名 |
| `AiSkill`、`AiTool`、`ToolRegistry` | `ai-runtime-contract` 中的内部运行时契约，短期保留技术语义 |
| `/api/skills` 兼容路径 | 若仍由 Controller 暴露，先保留兼容入口，产品文档主推 `/api/compositions` 或 `/api/capabilities` |
| `ai-skills-service` artifactId / 目录名 | 本轮不改变 Maven 模块和部署单元 |

如果未来决定“不考虑旧数据兼容”并进入 SQL/API 迁移阶段，也应单独建迁移计划，配套 `sql/init.sql`、upgrade SQL、实体、Mapper、前端类型和端到端验证。本轮不做这一步。

## 不改变的运行合同

为保证“不丢失现在的功能，不影响现在的页面展示和 UI”，本轮只做命名口径和文档/注释/UI 文案级调整。

必须保持：

- `ai-model-service`、`ai-skills-service`、`ai-agent-service` 三个 Maven module 和 artifactId 不变。
- 默认端口 `18601`、`18602`、`18603` 不变。
- 管理端代理 `/model`、`/ai`、`/api` 不变。
- 后端 Controller 路由不变。
- `sql/init.sql` 和 `sql/upgrade-*.sql` 不因本轮发生 schema 变化。
- Workflow Studio 页面结构、路由、GraphSpec、Agent binding 和 Runtime 执行链路不变。
- Dashboard 服务健康检查 key 仍可使用 `ai-skills-service`，但展示名可以显示为 Knowledge / Retrieval。

## 分阶段迁移建议

### 第一阶段：边界和语言收敛

当前阶段已经开始：

- 文档主线使用五个逻辑域解释架构。
- README、服务 README 和前端说明不再把 `ai-skills-service` 表述为 Skill 服务。
- 新增验证脚本检查边界文档、服务展示名和前端代理不变。

### 第二阶段：代码归属收敛

后续改动按逻辑域归位：

- 新增能力资产相关代码放入 Capability Catalog 边界。
- 新增 Workflow/Runtime/Debug/Human Approval 相关代码放入 Runtime Host 边界。
- 新增身份、ACL、Guard、MCP、A2A、Gateway 相关代码放入 Platform Control 边界。
- Knowledge/Retrieval 只承载知识、检索、RAG、业务索引和文档处理。

### 第三阶段：低风险包名整理

当一组文件没有外部 API、SQL、前端引用风险时，可做包名整理。例如把纯展示或纯文档相关类从旧 `skill` 口径迁到 `capability` 口径。每次迁移都要配套编译或目标测试。

### 第四阶段：物理服务拆分评估

只有当逻辑边界稳定后，才评估是否从 `ai-agent-service` 中拆出：

- `reachai-capability-service`
- `reachai-runtime-service`
- `reachai-control-service`

拆分时必须重新定义表所有权、服务间 API、事务边界、前端代理和部署拓扑。

## 验证

本轮重塑的基础验证入口：

```powershell
node scripts/check-backend-boundary-naming.mjs
git diff --check
cd ai-admin-front
npm run build
```

验证含义：

- `check-backend-boundary-naming.mjs` 确认五个逻辑域、Knowledge/Retrieval 口径和前端代理不变。
- `git diff --check` 确认文档和代码 diff 无空白错误。
- `npm run build` 确认前端文案调整不破坏当前页面构建。
