# 平台定位与架构总览

## 平台定位

ReachAI 是面向 Java 企业系统的 AI 能力中台。它不是单纯的 Workflow Builder，也不是只做历史项目扫描生成 Tool 的工具，而是围绕企业能力发现、注册、资产化、编排、执行、治理和开放形成一条可追踪的生产链路。

当前主线是业务系统通过 `reachai-spring-boot2-starter`、`reachai-capability-sdk`、`@ReachCapability` 和 `@ReachParam` 主动声明能力；平台侧形成能力快照、字段级 diff、评审、apply/ignore，并沉淀到 Capability Catalog；Workflow Studio 使用 `GraphSpec` 编排运行语义；Runtime Host 执行 Agent、Workflow、Trace、RunOps 和调试链路。

## 当前部署拓扑

第一阶段保持同一个 MySQL 库，不拆库。后端主路径已经进入物理服务拆分后的旧结构退场阶段，当前五个部署单元是：

| 部署单元 | 长期职责 | 默认端口 |
| --- | --- | --- |
| `reachai-control-service` | Platform Control public API/BFF，统一承接 `/api/**`、`/embed/**` 和 SDK 注册兼容入口 | `18603` |
| `reachai-runtime-service` | Runtime Host，承接 Agent、Workflow、GraphSpec、Trace、RunOps、调试和运行时内部 API | `18604` |
| `reachai-capability-service` | Capability Catalog，承接 SDK 注册、能力快照、diff/review/apply、扫描目录和能力资产 API | `18605` |
| `reachai-knowledge-service` | Knowledge / Retrieval，承接知识库、文件、Chunk、RAG、业务索引、向量检索和扫描器实现 | `18602`，context path `/ai` |
| `reachai-model-service` | Model Gateway，承接模型实例、Chat、Embedding、Rerank 和 OpenAI 兼容代理 | `18601` |

旧 `ai-agent-service` module 已从仓库主路径删除，不再作为 Maven、IDEA、本地启动或部署单元存在。剩余兼容路径必须迁入 owning service 的本地实现，或作为明确兼容桥记录后正式退场。

## 前端入口

`ai-admin-front` 是管理控制台。开发代理按路径进入当前服务：

| 前端路径 | 后端归属 |
| --- | --- |
| `/api/**` | `reachai-control-service` |
| `/ai/**` | `reachai-knowledge-service` |
| `/model/**` | `reachai-model-service` |
| Runtime / Capability 健康状态 | 通过 `reachai-control-service` 的 `/api/internal-services/health` 聚合 |

前端不直接调用 `reachai-runtime-service:18604` 或 `reachai-capability-service:18605` 的内部端口。

## 运行语义

`GraphSpec` 是 Workflow 的可执行语义，`canvas_json` 只表示画布布局。新增 Workflow Studio 节点、AI 编辑能力或 Runtime 行为时，必须把可执行语义写入 Workflow `GraphSpec`，不能只修改前端画布表现。

Runtime 执行主线位于 `reachai-runtime-service`。Capability 资产与业务系统 SDK 注册主线位于 `reachai-capability-service`。知识检索、RAG 和向量能力位于 `reachai-knowledge-service`。模型调用统一经过 `reachai-model-service`。

## 架构约束

- 公共 API 入口保持在 `reachai-control-service`，不要让前端重新认识旧大后端。
- 同库阶段仍要遵守服务表所有权和服务间 API 边界，不允许为了快速编译跨服务直接写对方表。
- 文档和产品口径默认使用 Capability / 能力；`Skill` 多为历史命名或兼容字段，不做盲目全局替换。
- SQL 基线入口是 `sql/initV2.sql`；后续数据库变更按需新增 `sql/upgrade-*.sql`，确认并入基线后可清理历史升级脚本。
